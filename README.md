# LiveKlass

## 프로젝트 개요

LiveKlass는 크리에이터(강사)가 강의를 개설하고 클래스메이트(수강생)가 강의에 수강 신청하는 수강 신청 API 서버입니다.

주요 사용 흐름은 다음과 같습니다.

- 사용자는 `강사(CREATOR)` 또는 `수강생(STUDENT)` 역할로 가입합니다.
- `강사`는 강의를 등록하고, 등록된 강의는 기본적으로 `DRAFT` 상태가 됩니다.
- 강의 소유자인 `강사`는 강의를 `OPEN` 상태로 변경해 모집을 시작하고, 필요 시 `CLOSED` 상태로 마감합니다.
- `수강생`은 `OPEN` 상태의 강의에만 수강 신청할 수 있으며, 신청 직후 상태는 `PENDING`입니다.
- 결제 확정 API를 호출하면 수강 신청 상태는 `CONFIRMED`가 되며, 결제 확정일이 저장됩니다.
- 수강 신청은 취소할 수 있고, 결제 확정 이후에는 확정일 기준 7일 이내에만 취소할 수 있습니다.
- 강의 정원은 `PENDING`, `CONFIRMED` 상태의 신청만 신청 인원으로 집계하며, `CANCELLED` 상태는 정원 계산에서 제외합니다.

## 요구사항 해석 및 가정

### 사용자(User)

- 별도의 인증/인가 서버는 구현 범위가 아니라고 보고, 요청 사용자는 `X-USER-ID` 헤더로 식별한다고 가정했습니다.
- 사용자 역할은 강의 개설 권한을 가진 `CREATOR`와 수강 신청 권한을 가진 `STUDENT` 두 가지로 단순화했습니다.
- 본 과제의 성격 상 사용자는 회원가입 및 유저 정보 조회를 단순화하였습니다.

### 강의(Course)

- 크리에이터(강사)는 강의를 개설하고 수강 정원, 가격, 기간을 설정합니다.
  - 강의는 `CREATOR` 역할의 사용자만 등록할 수 있다고 해석했습니다.
- 강의 상태는 `DRAFT`, `OPEN`, `CLOSED`로 관리합니다.
  - 강의 등록 직후 상태는 `DRAFT`이며, 강의 소유자인 `CREATOR`만 `OPEN`, `CLOSED` 상태로 변경할 수 있다고 해석했습니다.
- 강의 목록 조회는 상태 필터를 지원합니다.
  - 상태값이 없으면 모집 중인 강의를 우선 보여주기 위해 `OPEN`을 기본값으로 사용합니다.
- 강의 상세 조회의 현재 신청 인원은 실제 수강 확정 인원만이 아니라 정원 점유에 영향을 주는 `PENDING`, `CONFIRMED` 신청 수로 해석했습니다.

### 수강 신청(Enrollment)

- 클래스메이트(수강생)는 원하는 강의에 수강 신청을 합니다.
  - 수강 신청은 `STUDENT` 역할의 사용자만 할 수 있다고 해석했습니다.
  - `DRAFT`, `CLOSED` 상태의 강의는 신청할 수 없고, `OPEN` 상태의 강의만 신청할 수 있도록 했습니다.
- 정원이 초과되면 신청이 불가합니다.
  - 정원이 초과되어도 강의의 상태는 모집 마감으로 변경하지 않았으며, 모집 마감 처리는 강사가 처리하는 것으로 해석하였습니다.
  - 모집 마감 상태는 정원이 초과되는 사유 말고도 수강 기간, 강사의 준비 등 다양한 경우의 수가 있을 것으로 가정하였습니다.
- `STUDENT` 역할의 사용자만 수강 신청, 내 수강 신청 목록 조회, 결제 확정, 수강 취소를 할 수 있다고 해석했습니다.
- 외부 결제 시스템 연동은 구현하지 않고, 결제 확정 API 호출을 결제 완료 이벤트로 간주해 `PENDING` 신청을 `CONFIRMED`로 변경합니다.
- 수강 취소 가능 기간은 결제 확정일 기준 7일 이내로 고정했습니다. 결제 전인 `PENDING` 신청은 취소 가능 기간 제한 없이 취소할 수 있습니다.
- 정원 계산에서는 `PENDING`, `CONFIRMED` 신청만 포함하고, `CANCELLED` 신청은 제외합니다.

### 수강생 목록 및 페이지네이션

- 강의별 수강생 목록은 크리에이터 전용 기능으로 해석했고, 해당 강의를 생성한 `CREATOR`만 조회할 수 있도록 했습니다.
  - 요청자가 `CREATOR` 역할인지 먼저 확인한 뒤, 해당 강의의 `creatorId`와 요청자의 ID가 일치하는지 검증합니다.
- 수강생 목록에서는 결제가 완료된 실제 수강생을 보여주기 위해 `CONFIRMED` 상태만 조회합니다.
- 페이지네이션은 강의 목록 조회와 강의별 수강생 목록 조회에 적용했습니다.

### 정원 관리 규칙

수강 신청 시 정원 초과를 막기 위해 다음 순서로 처리합니다.

1. 신청 대상 강의를 조회하면서 해당 강의 row에 비관적 쓰기 락을 획득합니다.
2. 요청 사용자가 `STUDENT` 역할인지 검증합니다.
3. 강의 상태가 `OPEN`인지 검증합니다.
4. 해당 강의의 현재 신청 인원을 조회합니다.
5. 현재 신청 인원이 강의 정원 이상이면 수강 신청을 거부합니다.
6. 정원이 남아 있으면 `PENDING` 상태의 수강 신청을 생성합니다.

현재 신청 인원은 `PENDING`, `CONFIRMED` 상태의 수강 신청만 포함합니다. `CANCELLED` 상태의 신청은 정원 계산에서 제외되므로, 취소된 자리에는 다른 수강생이 다시 신청할 수 있습니다.

동시에 여러 사용자가 마지막 자리에 신청하는 경우를 고려해 강의 row 기준 비관적 쓰기 락을 사용합니다. 하나의 트랜잭션이 정원 확인과 신청 생성을 끝낸 뒤 다음 요청이 정원을 다시 확인하므로, 최종 신청 인원이 강의 정원을 초과하지 않도록 제어합니다.

## 설계 결정과 이유

- API, 애플리케이션, 도메인, 인프라 계층을 분리했습니다. Controller는 HTTP 요청/응답 변환에 집중하고, Facade는 유스케이스 조합과 권한 검증을 담당하며, Domain Service와 Entity는 핵심 비즈니스 규칙을 처리합니다.
- `Criteria`, `Command`, `Info`, `Result`, `Dto` 객체를 분리해 계층 간 데이터 전달 모델을 명확히 했습니다. 이를 통해 API 요청 모델이 도메인 내부 구현에 직접 의존하지 않도록 했습니다.
- 강의와 수강 신청의 상태 전이는 Entity 메서드로 캡슐화했습니다. 예를 들어 수강 신청 확정은 `confirm()`, 취소는 `cancel()`에서 처리해 상태 변경 규칙이 여러 계층에 흩어지지 않도록 했습니다.
- 정원 초과 방지를 위해 수강 신청 시 강의 row에 비관적 쓰기 락(`PESSIMISTIC_WRITE`)을 획득한 뒤 신청 인원을 다시 계산합니다. 동시에 여러 사용자가 마지막 자리에 신청하더라도 한 트랜잭션씩 정원 확인과 신청 생성이 진행되어 정원을 넘지 않도록 설계했습니다.
- 정원 계산에서는 `PENDING`과 `CONFIRMED`만 포함합니다. 결제 대기 중인 신청도 자리를 점유한다고 보아 중복 초과 신청을 막고, 취소된 신청은 즉시 정원에서 제외해 다른 수강생이 신청할 수 있도록 했습니다.
- Repository는 도메인 인터페이스와 JPA 구현체를 분리했습니다. 도메인 계층은 저장소의 구체 기술을 모르고, 인프라 계층에서 Spring Data JPA Repository를 감싸도록 구성했습니다.
- 공통 응답 포맷(`ApiResponse`)과 전역 예외 처리(`ApiControllerAdvice`)를 사용해 성공/실패 응답 구조를 통일했습니다. 비즈니스 예외는 `CoreException`과 `ErrorType`으로 HTTP 상태와 메시지를 일관되게 매핑합니다.

## 미구현 / 제약사항

- 대기열 기능은 구현하지 못했습니다. 
- 강의 정원이 초과된 경우 별도의 대기 상태로 등록하지 않고, 수강 신청을 거부하였습니다.

## AI 활용 범위

본 과제의 핵심 구현, 아키텍처 설계, 도메인 규칙 정의, 테스트 작성은 직접 수행했습니다.

AI는 주로 이미 작성한 코드 구조와 테스트 패턴을 바탕으로 반복적인 뼈대 코드 작성, 단순 기능 구현, 테스트 보강에 제한적으로 활용했습니다.  

특히 계층 간 단방향 의존성, 도메인과 인프라 분리, 일관된 네이밍과 테스트 스타일을 유지할 수 있도록 참고 도구로 사용했습니다.

AI가 생성한 코드도 그대로 사용하지 않고, 요구사항과 아키텍처 규칙에 맞는지 직접 검토하고 수정했습니다.  

즉, AI는 구현을 대체하기보다 기존에 직접 설계하고 작성한 코드 품질을 유지하면서 반복 작업을 보조하는 용도로 활용했습니다.
